Pattern Matching!

Make sure you've read up on Variant first.

Por fin llegamos aquí! Coincidencia de patrón (Pattern matching) es una de las mejores características de el lenguage. Es como la desestructuración, pero viene con todavía más ayuda del sistema.

Uso

Considera la variante:

type payload =
  | BadResult(int)
  | GoodResult(string)
  | NoResult;

Mientras uses la expresión switch en ella, puedes "destructurarlo":

data = GoodResult("¡Product expedido!");

let message =
  switch (data) {
  | GoodResult(theMessage) => "¡Triunfo! " ++ theMessage
  | BadResult(errorCode) => "Algo fue mal. El código de error es: " ++ string_of_int(errorCode)
  };

Observa cómo hemos desestructurado data mientras se gestiona cada caso diferente. El anterior switch te dará una advertencia del compilador:

Warning 8: this pattern-matching is not exhaustive.
Here is an example of a value that is not matched:
NoResult

¿No es genial? Mientras se empareja con la forma de tus datos, el sistema de tipos te advirtió de un caso no controlado. Este aspecto condicional es lo que hace de la coincidencia de patrón (Pattern matching) diferente de la desestructuración plana. La mayoría de las estructuras de datos con un aspecto "si esto entonces aquello" funcionaran con ella:

switch (myList) {
| [] => print_endline("Lista vacia")
| [a, ...theRest] => print_endline("lista con el valor de cabeza " ++ a)
};

switch (myArray) {
| [|1, 2|] => print_endline("Esto es un array con el item 1 y 2")
| [||] => print_endline("Este array no tiene elementos")
| _ => print_endline("Esto es un array")
};

El caso _ es un caso especial (fall-through) que permite a todas las condiciones que no coinciden ir a esa rama.

Incluso puedes usar como de condición una cadena, int y otros. ¡Incluso puedes tener muchos patrones dirigidos al mismo resultado!

let reply =
  switch (message) {
  | "Reason es geniall" => "Yep"
  | "buenas noches" => "¡Nos vemos!"
  | "hola" | "hi" | "heya" | "hey" => "¡hola a ti tambien!"
  | _ => "¡Encantado de conocerte!"
  };

Combinado con otras estructuras de datos, la coincidencia de patrón puede producir código extremadamente conciso, verificado por el compilador, de alto rendimiento:

let message =
  switch (data) {
  | GoodResult(theMessage) => "¡Triunfo! " ++ theMessage
  | BadResult(0 | 1 | 5) => "Algo fue mal. Es un problema de lado del servidor."| BadResult(errorCode) = > "ocurrió un error desconocido. Code: " ++ string_of_int(errorCode)
  | NoResult => "Todo parece correcto"
  };

Nota: sólo puedes pasar literales (es decir, valores concretos) como un patrón, no nombres de vínculos let (let-binding) u otras cosas. Lo siguiente no funciona como se esperaba:

let myMessage = "Hola";
switch (greeting) {
| myMessage => print_endline("Hola a ti")
};

En cambio, asumiría que estas emparejando con cualquier cadena de caracteres y vinculandolo con el nombre myMessage en ese caso switch, que no es lo que querías.

Cláusulas "Where"

Cuando realmente necesitas utilizar una lógica arbitraria con un, de alguna manera limpio, patrón de coincidencia, puedes introducir algunas cláusulas when, que son básicamente endulzamiento (sugar) if:

let message =
  switch (data) {
  | GoodResult(theMessage) => ...
  | BadResult(errorCode) when isServerError(errorCode) => ...
  | BadResult(errorCode) => ... /* si no */
  | NoResult => ...
  };

Coincidencia en las excepciones

Si la función genera una excepción (tratado más adelante), también puedes hacer coincidir en ella, además de los valores normalmente devuelto por la función.

switch (List.find((i) => i === theItem, myItems)) {
| item => print_endline(item)
| exception Not_found => print_endline("¡Tal item no encontrado!")
};

Patrones anidados

Los | anidados funcionan según lo previsto:

switch (student) {
| {name: "Jane" | "Joe"} => ...
| {name: "Bob", Job: Programmer({fullTime: Yes | Maybe})} => ...
};

Patrones por todas partes

Puedes poner un patrón en cualquier lugar que pondrías una "declaración de variable" normal:

type leftOrRight =
  | Left(int)
  | Right(int);

let i = Left(1);

/* magic! */
let Left(v) | Right(v) = i;

Consejos y Trucos

Elimina los anidamientos de tu patrón de coincidencia siempre que puedas. Se trata de una manera de eliminar fallos eficaz. Ejemplo a continuación.

No abuses de el caso fall-through _ demasiado. Esto impide que el compilador te diga que has olvidado cubrir un caso (control exhaustivo), que sería especialmente útil después de una refactorización donde agregas un nuevo caso a una variante. Intenta utilizar _ sólo frente a infinitas posibilidades, por ejemplo, string, int, etc.

Aquí tienes una serie de ejemplos, de peor a mejor:

let optionBoolToBool = opt =>
  if (opt == None) {
    false;
  } else if (opt == Some(true)) {
    true;
  } else {
    false;
  };

Esto es simplemente absurdo =). Vamos a convertirlo en un patrón de coincidencia:

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(a) => a ? true : false
  };

Un poco mejor, pero todavía están anidados. La coincidencia de patrón te permite hacer esto:

let optionBoolToBool = opt =>
  switch (opt) {
  | None => false
  | Some(true) => true
  | Some(false) => false
  };

¡Un aspecto mucho más lineal! Ahora, puedes estar tentado a hacer esto:

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(true) => true
  | _ => false
  };

Which is much more concise, but kills the exhaustiveness check mentioned above; refrain from using that. This is the best:

let optionBoolToBool = opt =>
  switch (opt) {
  | Some(trueOrFalse) => trueOrFalse
  | None => false
  };

¡Rematadamente difícil cometer un error en este código, en este momento! Cuando desees utilizar un if-else con múltiples ramas, elige patrones de coincidencia en su lugar. Es más conciso y eficiente también.

Decisiones de Diseño

El famoso problema fizzbuzz curiosamente hace tropezar a algunas personas, en parte debido a su naturaleza de paralizar el programador que espera simplificar/unificar las pocas ramificaciones condicionales en busca de elegancia donde no la hay. Ojalá puedas ver que generalmente, la simplificación visual de la coincidencia de patrón nos permite superar la parálisis de decisión, mientras mantiene todos los beneficios (y más, como has visto) de un montón de if-else a fuerza bruta. No hay nada de malo con listar explícitamente todas las posibilidades; La coincidencia de patrón se corresponde al análisis de caso en matemáticas, una técnica válida para la resolución de problemas que demuestra ser muy practica.

Usando un switch de Reason por primera vez podría hacerte sentir como que te lo has estado perdiendo todos estos años. Cuidado, porque eso podría estropear la opinión que tienes de otros lenguajes =).

Si has probado a refactorizar la lógica de un if-else grande y anidada podrías darte cuenta que es muy difícil conseguir la lógica correcta. Por otro lado, coincidencia de patrón + tupla conceptualmente mapea a una tabla 2D, donde cada celda puede ser rellenada independientemente. Esto asegura que cada vez que necesites añadir un caso en el switch, puedes apuntar sólo a esa celda de la tabla, sin estropear otras celdas.

type animal = Dog | Cat | Bird;
let result = switch (isBig, myAnimal) {
| (true, Dog) => 1
| (true, Cat) => 2
| (true, Bird) => 3
| (false, Dog | Cat) => 4
| (false, Bird) => 5
};
